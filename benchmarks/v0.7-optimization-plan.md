# Shrinkage v0.7 — optimization & tuning plan

> **Status (v0.8.0):** shipped — economy tiering (v0.7), inline gate/score with
> explicit no-subagent instruction, lite-gate for small changes, rules-once
> per session, script-only score, low-reuse awareness, gate ledger
> (gatelog.py + diffstat cross-check), compat-watch (signature changes),
> shave --dry-run, DEPRECATIONS.md ledger, coverage-joined audit signal,
> T0 auto-PR bot + badge (ci-integration.md), full pytest suite + CI, and the
> five script bug fixes (deletion-aware refresh, rename-safe diffstat,
> comment-stripped refcounts, api-map merge, string-safe brace scanner).
> Subagent mechanism confirmed from docs: skills run INLINE by default;
> `context: fork` is opt-in and we never set it — bake-off subagent usage was
> model-chosen delegation, now explicitly instructed against.
> Remaining: the live /cost verification re-run against the $15.45 baseline.

Grounded in the greenfield bake-off (`greenfield-crypto-bakeoff.md`). The core
diagnostic: in the Shrinkage-only run, **output tokens were within 5% of vanilla
(39.9k vs 38.0k) but cache reads were 3.5× (7.7M vs 2.2M).** Shrinkage doesn't
make the model *generate* more — it makes it *read* more. Every lever below
targets context-reading, not behavior.

Cost attribution from `/cost`: `/srk:gate` ≈ 8% and `/srk:score` ≈ 4% of usage,
both appearing as **subagents** (srk:gate 8%, srk:score 3%). srk plugin = 23% of
the SHK-only session. That's the budget to reclaim.

## Priorities (by expected token savings)

### 1. Make `/srk:score` a pure script report (no subagent, no reasoning)
`score` runs `diffstat.py` and reports one line — that's mechanical, it needs no
model judgment and no isolated context. Today it shows as a ~3–4% subagent that
re-loads context to say a line the script already printed. Fix: the command runs
the script and echoes the line verbatim, inline. **~3–4% reclaimed.**

### 2. Run `/srk:gate` inline, and add a "lite gate" for small changes
Two problems: (a) gate runs as a subagent (8% + 9% subagent) that re-reads the
codemap + rules cold, and (b) it runs the *full* ceremony (5 candidates, ladder
walk, catalog-in-reverse, compat pass, gate record) even for a one-line change
like "add a Sortino metric."
Fix: gate reasoning happens in the **main context** (where the map and rules are
already loaded/cached), and scales to the change: a **lite path** — quick codemap
check for an obvious owner, proceed — for small/low-risk edits, reserving the full
ceremony for changes above a size/impact threshold or when the map flags a likely
duplicate. **Biggest single lever — most of the ~11–17% overhead.**

> Prerequisite / honest unknown: confirm *why* gate/score run as subagents. If our
> command/skill spawns them, remove the spawn. If Claude Code runs every plugin
> slash command in a sub-context by default, we instead (a) make each command read
> less per call, and (b) push the gate discipline into the model-invoked **skill**
> (which runs in the main context) so routine tasks never spawn a per-task
> subagent — the slash command becomes the explicit-use path, not the default.

### 3. Load doctrine once per session, not per call
The 3.5× cache reads come from re-reading `rules/<lang>.md`, workflow files, and
refs on repeated gate/score calls. Fix: load `rules/<lang>.md` once (at onboard /
first gate) and reference thereafter; **trim gate's required reading** — gate
should NOT pull `safety-model.md` (that's for deletion/shave, not adding code).
Keep the codemap access targeted (`query`, not full-map re-reads).

### 4. Cheaper models for mechanical subagents
The cost report literally suggests it, and 82–94% of usage was subagent-heavy.
Fix: audit auditor subagents → **Haiku** via agent frontmatter (they're read-only
pattern scans; matters for audit-heavy sessions). Score → script (no model).
Gate's candidate-harvest step is a codemap query (mechanical).

### 5. Self-awareness on low-reuse repos
The skill's own description says it's "not needed for brand-new empty projects,"
and greenfield showed streak 0 / all growth. Fix: when the codemap has few symbols
(empty/near-empty repo), note "reuse gate adds little here — running lite" instead
of full ceremony. Don't pay for reuse machinery when there's nothing to reuse.

## Tuning fixes (quality, already done or specified)

- **Scoreboard accuracy** (DONE, proven): `diffstat.py` no longer counts docs/
  config as code — `SPEC.md`/`.json` were inflating "app LOC" (213 → true 204).
  Fixture-tested before/after + regression guard.
- **Pinned walk-forward windows** (DONE in SPEC): train 2000 / test 500 / step 500
  so re-runs are numerically comparable across arms.
- **Configurability signal** (optional): the walk-forward comparison showed
  SHK-only built fewer knobs (no `--train/--test`) — the doctrine working, but
  sometimes you want the knob. The gate could surface "this looks like something a
  user will configure — consider a parameter now" rather than pure minimalism.

## How we prove it worked

Efficiency is a **runtime** property — it can't be unit-tested in a sandbox, only
measured live. The test is a controlled re-run:

1. Ship v0.7.
2. Rebuild **arm C** (Shrinkage-only) on the pinned SPEC.
3. Compare `/cost` to the **$15.45 baseline**; confirm the code stays tight
   (~204 app LOC, ~108 statements, 99% coverage, acceptance green) so it got
   cheaper *without* getting worse.
4. Run it twice to beat single-run noise.

Target: pull Shrinkage-only from ~$15 toward **~$8–10** — closing most of the gap
to vanilla's $6 while keeping the half-the-code win.

## What NOT to change

- The safety model, evidence chain, and one-transform-per-commit rules — those are
  correctness, not overhead, and the deletion machinery wasn't even exercised here.
- The extension ladder / anti-speculation doctrine — it's what produced the
  half-the-code win. The goal is to deliver that discipline *cheaper*, not to dilute it.
